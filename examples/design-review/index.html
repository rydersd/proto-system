Making the dashboard make sense.

Two weeks, in one paragraph.

We started from the support-ticket review — three of the top ten themes traced back to the reporting dashboard being hard to read. So the fortnight became a focused pass on Beacon's reporting surface: a metric drill-down that opens the rows behind any number, a saved-views model so a team can keep its own slice of the data, and an empty-state sweep so a fresh workspace never shows a blank panel. Underneath all of it, an unsettled question — what the default chart should be when a user has not chosen one — which we could not close alone.

The conversations that set the agenda.

Three inputs from outside the design team shaped what we worked on. Some settled, some still open. Worth naming each so the rest of this review lands in context.

Two weeks ago · Support sync

Top-ten ticket themes

The support team's monthly review surfaced that "I can't tell what this number means" is one of the most common reasons people open a ticket. That anchored the agenda.

Last week · Product ask

Make the dashboard self-serve

The brief: a new user should be able to read the reporting surface without a walkthrough. Reduce the questions before they reach support.

Last week · Data team

Chart-default refresher

A refresher with the data team on what each metric actually measures. It confirmed the importance of a sensible default chart — and surfaced that no one owns that decision yet.

Ongoing · Engineering

Saved-views feasibility

Early feasibility check with engineering: saved views can ship without a schema change if they are stored per-user. Confirmed, and it shaped the design.

Every number opens.

The core change: a metric is no longer a dead end. Click any KPI tile or chart segment and the rows behind it slide in — the same records, filtered to exactly that number. A chart that cannot be opened is a claim you have to take on faith.

Two asks. Each scoped.

If you only do one thing in this review — do these.

Ask 01 · Decision needed

Default chart ownership

The data team confirmed each metric needs a sensible default chart, but no one owns that call. We need an owner and a date — otherwise the drill-down ships against an inconsistent baseline.

Want by: next review · Blocks: drill-down rollout
Ask 02 · Review & sign-off

Saved-views model — is per-user right?

Per-user storage avoids a schema change but means views are not shareable. Need a yes / no / "make them shareable later" from product before we wire it.

Want by: end of week · Format: 20-min walkthrough

Honest list. Things we can't move alone.

Each of these depends on someone outside the design team.

Default chart per metric

The drill-down works, but the baseline chart it opens against is inconsistent. Needs a data-team owner. Tracked as Ask 01.

Shareable views

Per-user views ship fast; team-shared views need a small schema addition. Trade-off call belongs to product. Tracked as Ask 02.

Historical data depth

Drill-downs older than 90 days are slow because the warehouse archives. A performance call for engineering, not design.

Mobile reporting

The reworked surface assumes a wide viewport. Whether reporting needs a real mobile layout is a roadmap question.

What deserves the next sprint.

Provisional ordering. The sprint commit is the conversation, not the spec.

Wk 1

Default-chart decision follow-through

Confirm the owner from Ask 01, then lock the baseline chart for each metric so the drill-down has a stable foundation.

Wk 1–2

Saved-views build (conditional on Ask 02)

If per-user is signed off, wire the view switcher end to end.

Wk 2–3

Empty-state rollout

Extend the empty-state pattern from reporting to the remaining workspace surfaces.

Throughout

Support-theme tracking

Watch whether dashboard tickets drop after the drill-down ships — the real measure of whether this fortnight worked.
