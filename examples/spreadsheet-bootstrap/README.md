# Spreadsheet Bootstrap — Nib Example

A complete Nib project scaffolded from a single workbook. Demonstrates every workbook tab type including a 3-flow nested service blueprint (program → deal-reg + onboarding) for the leadership bird's-eye / drill-down view.

## Files

```
examples/spreadsheet-bootstrap/
├── example.xlsx               # The source of truth
├── build-example-workbook.js  # Regenerates example.xlsx
├── index.html                 # Demo page (loads everything from data/)
├── data/                      # ← Generated by nib-ingest. Do not edit by hand.
│   ├── wireframe-config.js
│   ├── sections.js
│   ├── tokens.css
│   ├── personas.js
│   ├── stories.js
│   ├── blueprints/
│   │   ├── _index.js          # Blueprint hierarchy (drives Overview drill-down)
│   │   ├── program.js
│   │   ├── deal-reg.js
│   │   └── onboarding.js
│   ├── registries/
│   │   └── initiatives.js
│   └── source-of-truth.txt    # Idempotent sidecar for nib-sync
```

## Round-trip

```sh
# 1. Regenerate the example workbook from the script
node examples/spreadsheet-bootstrap/build-example-workbook.js

# 2. Ingest the workbook into a Nib project
node tools/nib-ingest.js examples/spreadsheet-bootstrap/example.xlsx \
  --out examples/spreadsheet-bootstrap

# 3. Open the demo
open examples/spreadsheet-bootstrap/index.html

# 4. Edit any tab in example.xlsx, then re-sync (only changed files are rewritten)
node tools/nib-sync.js --project examples/spreadsheet-bootstrap
```

## Worked tabs

| Tab | Purpose | Key cells |
|---|---|---|
| `meta` | WIREFRAME_CONFIG | title, defaultSurface, feedbackEndpoint, portalHeader, search |
| `pages` | SECTIONS + page tree | id/label/parent/surface/template/blueprintId |
| `tokens` | CSS variable overrides | Two-column rows; bare names get `--wf-` prefix |
| `personas` | STORY_MAP + persona cards | jtbd/pains/goals are `;`-separated |
| `stories` | DESIGN_STORIES + STORY_TITLES | kind = `jtbd` or `design` |
| `program` | Root service blueprint | meta:parent left blank → root; meta:summary + meta:whatChanges feed the leadership Overview card |
| `deal-reg` | Child blueprint | meta:parent = `program`; reachable via drill-down from program nodes with childBlueprintId |
| `onboarding` | Sibling child blueprint | Same pattern as deal-reg |
| `_initiatives` | Registry (controlled vocab) | Underscore-prefixed → emitted to data/registries/initiatives.js |

## Nested-blueprint pattern

The `program` tab demonstrates the leadership bird's-eye view:

- Top-level `meta:summary` + `meta:whatChanges` rows feed the Overview card visible to leadership.
- Several nodes carry a `childBlueprintId` value (`onboarding`, `deal-reg`) — those nodes drill into a sub-blueprint when clicked in the canvas.
- Sub-blueprints (`deal-reg`, `onboarding`) declare `meta:parent: program` so the breadcrumb knows how to walk back up.

Open `index.html` to see the rendered hierarchy. The full editable React Flow canvas ships in `examples/service-blueprint/` (Track 2).
