# Creating a Project

> ⚠️ **Deprecated.** Edit [`docs/Create-Project.md`](../docs/Create-Project.md) instead. This file will be removed in a future release.

> Read when scaffolding a brand-new Nib project. Replaces "copy a starter" for most cases.

## One-line setup

```sh
npx create-nib lead-to-cash
```

(Once nib is published — see "Distribution" below for the path while it's still local.)

The CLI:

1. Creates `./lead-to-cash/` (refuses if non-empty).
2. Boots a local HTTP server on `5173`.
3. Opens your browser to `http://127.0.0.1:5173/?name=lead-to-cash`.
4. The wizard offers three paths: **upload a workbook**, **pick a template**, or **start blank**.
5. On submit, the server runs `nib-ingest` (workbook) or copies the template directory; then redirects to `/project/index.html`.
6. The server stays up so you keep editing in the browser. `Ctrl-C` to stop.

The wizard derives a Title-Cased project title from the kebab name (`lead-to-cash` → `Lead To Cash`) and stamps it into `WIREFRAME_CONFIG.title`.

## Headless flags

Skip the wizard if you already know what you want:

```sh
# Spawn from a workbook — uploads + ingests, no browser
npx create-nib partner-program --workbook ./pcp.xlsx

# Spawn from a Google Sheet (sharing URL → /export?format=xlsx)
npx create-nib q3-research --sheet https://docs.google.com/spreadsheets/d/<id>

# Start from a template, no questions asked
npx create-nib service-design --template service-blueprint
```

| Flag | Meaning |
|---|---|
| `--template <id>` | Skip picker; copy `examples/<id>/` |
| `--workbook <path>` | Skip picker; ingest local xlsx |
| `--sheet <url>` | Skip picker; ingest Google Sheets URL |
| `--port <n>` | Wizard port (default 5173) |
| `--no-open` | Don't auto-open the browser |
| `--target <dir>` | Override target dir (default `./<name>`) |

## What the three wizard paths produce

### 📊 From a workbook
- File picker accepts `.xlsx`; alternatively a Google Sheets URL field.
- Server saves the file into the project as `<filename>.xlsx` (the source of truth — `nib-sync` reads from there on subsequent runs).
- Runs `nib-ingest` against it.

### 🎨 From a template
- Lists every directory under `examples/` with the first paragraph of its README.
- Server runs `copyDirSync(examples/<id>, <target>)` then patches `data/wireframe-config.js` `title` field to match the project name.

### ✨ Start blank
- Writes a minimal `project-data.js` (one section, one page) and a one-page `index.html` linking the framework's CSS.
- Useful when you want to add pages by hand without committing to a template shape.

## Server endpoints

The embedded server exposes a small API the wizard uses:

| Endpoint | Method | Purpose |
|---|---|---|
| `/` | GET | Wizard UI |
| `/api/templates` | GET | `{ templates, suggestedTitle }` |
| `/api/setup` | POST | `{ mode: 'workbook'\|'template'\|'blank', ... }` → runs setup, returns redirect URL |
| `/project/*` | GET | Serves the new project's static files (after setup) |
| `/nib/*` | GET | Serves the nib framework so the project's `<link>`/`<script>` tags resolve |

The server only binds to `127.0.0.1` — never exposes the file system to the network.

## What lands in your new project

```
lead-to-cash/
├── index.html              # Entry point (paths rewritten to nib/core/...)
├── data/                   # Generated by ingest (when scaffolding from a workbook)
├── example.xlsx            # The source workbook (when applicable — re-runnable via nib-sync)
└── nib/                    # Bundled framework subset — makes the project self-contained
    ├── core/               # CSS + JS runtime
    ├── surfaces/           # Slack / Salesforce / Internal overlays
    └── starters/           # Reference HTML stubs
```

`tools/`, `examples/`, and `node_modules/` from the framework repo are *not* copied — only what's needed at runtime. The bundled `nib/` directory means `python3 -m http.server` from the project root Just Works; no symlinks, no path tricks.

## Distribution

Two packages live in the repo:

```
nib/                                # Framework package
├── package.json                    # name: nib, bin: nib-ingest / nib-sync / nib-create
└── tools/nib-create.js             # The actual scaffolder

packages/create-nib/                # Thin wrapper for `npm create nib`
├── package.json                    # name: create-nib, deps: nib ^0.3.0
└── bin.js                          # require('nib/tools/nib-create.js')
```

To publish (when ready):

```sh
# 1. Publish the framework
cd /path/to/nib
# Flip "private": true → false in package.json
npm publish

# 2. Publish the create wrapper
cd packages/create-nib
npm publish
```

Both `npm create nib <name>` and `npx create-nib <name>` then resolve. `npx nib-create <name>` also works directly off the framework package.

## Local development (before publishing)

Run the scaffolder directly from the nib repo:

```sh
node tools/nib-create.js my-project
```

To make it global (so any directory can run `nib-create my-project`):

```sh
cd /path/to/nib && npm link
nib-create my-project   # works anywhere
```

To exercise the wrapper package locally:

```sh
cd packages/create-nib && npm install ../..
node bin.js my-project
```

`npm unlink nib` reverses `npm link`.

## Local development (before publishing)

From inside the nib repo:

```sh
node tools/nib-create.js <project-name>
```

To make it available globally (so you can run `nib-create <name>` from any directory):

```sh
cd /path/to/nib
npm link
nib-create my-project
```

`npm link` symlinks `bin.nib-create` and `bin.create-nib` into your global `node_modules`. Run `npm unlink nib` to undo.

## Why a server, not a static file?

The wizard could run as a `file://` HTML page — but:
- **File uploads** would have to be parsed in the browser then written back somehow (no filesystem access from `file://`).
- **ES module imports** (the React Flow canvas) are blocked from `file://` in most browsers.
- The wizard already runs `nib-ingest` server-side, so we need Node anyway.

Embedding the server costs ~250 lines and gives a smooth, modern flow with a single `npx` command.
